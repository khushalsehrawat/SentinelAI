Flow

How SentinelAI thinks like a SOC analyst

Under the hood, the project follows a clear threat-intelligence pipeline — exactly what hiring managers expect from a real SOC or SIEM product.

1. Ingest logs & errors

Apps, services, dev tools or security appliances send logs to /api/logs or through the UI (paste / upload). Each entry is normalized into a LogEntry with source, title, raw text and metadata.

2. AI enrichment

LogService calls AIService, which runs the log through a GenAI model. It returns severity, category, threat score, MITRE technique and recommendation. Validation ensures safe default values when AI is unavailable.

3. Incident creation & correlation

Using AI output, the platform creates an Incident object that stitches together summary, techniques, recommendations and the related log. This becomes the core unit for SOC investigation.

4. Alerts & auto-action suggestions

When severity is HIGH / CRITICAL, AlertService triggers an email notification to admins and logs any suggested auto-actions (BLOCK_IP / ISOLATE_HOST) for integration into real security systems.

5. Analyst UI + AI side-chat

The dashboard shows enriched logs with severity chips, AI explanations and an AI "side-chat" to ask follow-up questions like “Explain this error in simple language” or “How do I fix this in Spring Boot?”.

Built like a production-grade backend

SentinelAI is structured for real-world maintainability: layered architecture, security best practices, and clear separation between ingestion, AI analysis and alerting responsibilities.

  • Spring Boot REST APIs for logs, incidents, alerts & users.
  • Service layer orchestration in LogService + AIService.
  • JPA entities: LogEntry, Incident, Alert, User.
  • Spring Security + JWT for authenticated, role-based access.
  • Email notifications wired through AlertService.
